# Shapes Rework Design
_Date: 2026-05-16_

## Summary

Expand BallBouncer's outer shape and inner cross-shape libraries from 16+12 to 24+18, replacing boring regulars with abstract-geometric and character wild shapes, fixing the broken custom-shape icon renderer, and reducing button height to accommodate the extra row without scrolling.

---

## 1. Layout

**Shape button height:** 28px ‚Üí 22px (gap stays 3px). This shrinks each grid by ~12px per row, freeing ~60px total ‚Äî exactly what adding one row to each grid costs, keeping the panel scroll-free at typical screen heights.

**Outer grid:** `repeat(8, 1fr)` √ó 3 rows = **24 shapes**
**Inner grid:** `repeat(6, 1fr)` √ó 3 rows = **18 crosses**

No panel width change. No scroll introduced.

## 4. Bug Fix: Custom Shape Icons

**Problem:** `iconPoints(shape, R, ox, oy)` destructures `{ sides, star, ir, sa }` ‚Äî properties not present on `custom:true` shapes. `sides` is `undefined`, `Array.from({ length: undefined })` returns `[]`, so the SVG `<polygon>` gets no points and renders as an invisible element. All 5 existing custom shape buttons appear blank.

**Fix:**
```javascript
function iconPoints(shape, R, ox, oy) {
    if (shape.custom) {
        // Scale normalized pts (‚àí1..1) to icon coordinate space
        return shape.pts.map(([x, y]) =>
            `${ox + x * R * 0.85},${oy + y * R * 0.85}`
        ).join(' ');
    }
    // existing polygon/star logic unchanged
    const { sides, star, ir = 0.4, sa } = shape;
    ...
}
```

Same fix applies to `iconCrossPoints` for custom cross shapes (Snowflake, Pinwheel, Star-Cross): detect `shape.custom`, render via `shape.pts` instead of the arm formula.

## 5. Custom Shape Point Coordinates

All coordinates are normalized to approximately ¬±1.0 range. The drawing functions scale by `SHAPE_R`. The icon function scales by `R * 0.85` to leave a small margin.

### Gear (8 teeth, 32 pts ‚Äî computed, stored as static array)
Generated by alternating outer radius 1.0 and inner radius 0.72 at tooth edge angles. Formula per tooth i (0..7):
- valley-left: `(i/8 - 1/32) * 2œÄ ‚àí œÄ/2`, r=0.72
- rise-right: `(i/8 + 1/32) * 2œÄ ‚àí œÄ/2`, r=0.72  
- tip-left: `(i/8 + 2/32) * 2œÄ ‚àí œÄ/2`, r=1.0
- tip-right: `(i/8 + 3/32) * 2œÄ ‚àí œÄ/2`, r=1.0

### Snowflake (custom cross, ~36 pts)
6-arm snowflake: each arm has a main shaft and two secondary branches at 60% length:
- Main arm tip at r=1.0
- Branch tips at r=0.65, ¬±30¬∞ from arm angle
- Valley between arms at r=0.15
Computed at module init into a static `pts` array using a helper function ‚Äî stored once, not regenerated per frame.

### Pinwheel (custom cross, 16 pts)
4 arms, each swept 30¬∞ clockwise from radial ‚Äî creates rotation illusion:
- Each arm is a parallelogram: near-left, far-left, far-right, near-right
- Arm length 1.0, width 0.22, rotation offset 30¬∞
Also computed at module init.

### Star-Cross (custom cross, 16 pts)
4 arms that taper to narrow shafts then flare into diamond-point tips:
- Shaft width 0.15, tip width 0.55 at 85% of arm length
- 4 points, each with 4 pts (narrow‚Üíwide‚Üítip‚Üíwide‚Üínarrow)
Also computed at module init.

## 6. Data Model

No new fields needed. `custom:true` shapes already support `pts` arrays. The existing `drawShape` function handles custom pts via its existing polygon-drawing path.

New cross custom shapes need to be handled in `drawCrossShape` the same way ‚Äî detect `shape.custom`, iterate `shape.pts`, draw polygon instead of arm formula.

---

## 7. What Does NOT Change

- Physics collision detection (polygon vertices used as-is)
- Ball/theme/audio system
- Panel structure, slider layout, color swatches
- `SHAPE_R`, `INNER_R` calculation
- Spin physics, trail rendering

---

## 8. Out of Scope

- Animated shape morphing
- User-defined custom shapes
- Per-shape color overrides
